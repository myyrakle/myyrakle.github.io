[원본 링크]

statefulset은 파드들의 상태(state) 관리에 치중된 컨트롤러 단위다.
deployment의 특수화된 변종이라고 보면 된다.

주로 데이터베이스 등을 구성할 때 사용한다.




Deployment의 문제

deployment는 상태가 없는 서버를 관리하기 위해 만들어진 구조라서, 상태가 필요한 DB 등의 케이스에 적용한다면 여러모로 좀 이상해질 수 있다.

별다른 설정을 하지 않고 deployment 같은 파드 그룹으로 프로비저닝을 하면, 파드가 뜨는 순서는 무작위다. 두번째 파드가 먼저 뜰 수도 있고, 첫번째 파드가 먼저 뜰 수도 있다.
일반적인 서버 환경에서는 이런게 문제가 되지 않는다.

하지만 데이터베이스 같은 분산시스템을 클러스터로 구성하거나 Replication하는 경우에는 문제가 발생한다.
예를 들어 master/slave 구조의 클러스터를 구성한다면, master 노드가 먼저 뜨고 그 이후에 slave 노드가 거기에 연동되는 형태가 된다. 당연히 master가 먼저 떠야할 것인데, slave가 먼저 뜬다면 참 곤란한 상황에 처할 것이다.

master/slave가 아니라 raft 같은 평등한 클러스터를 구성하더라도 문제가 사라지진 않는다. 순서에 대한 중요성은 낮아지지만 네트워크 수준 안정성이 중요해지기 때문이다.
예를 들면, raft는 네트워크 피어(노드) 단위로 실시간으로 합의를 하고 그걸 디스크에 저장하면서 데이터 무결성을 유지하는데, Pod가 교체되면서 네트워크가 교체된다면 그 가정이 심각하게 훼손되기 때문이다.

분산시스템의 구현 패턴에 따라서 문제가 드러나는 형태가 달라질 수는 있는데, 아무튼 문제는 이런 deployment의 유연한 동작 패턴이 분산시스템 구현체들과 잘 맞지 않는 부분들이 많다는 것이다.




StatefulSet의 기능

그래서 Statefulset은 다음과 같은 부가기능을 제공한다.

  1. Pod마다 고유한 이름을 부여할 수 있다.
  2. Pod마다 고유한 DNS를 부여할 수 있다.
  3. Pod마다 고유한 PV(볼륨)을 부여할 수 있다.
  4. Pod들의 생성/종료 순서를 index 번호 기반으로 강제로 직렬화한다.



사용해보기 - MySQL Statefulset 만들어보기

MySQL DB를 띄우는 간단한 예제를 돌려보자.

Statefulset을 제대로 사용하기 위해서는 먼저 Service가 필요하다.
그래야 고정된 DNS 이름을 생성할 수 있고, peer 연결 등을 제대로 할 수 있기 대문이다.

apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None
  selector:
    app: mysql
  ports:
    - port: 3306

적당히 만들어주고

거기에 맞춰서 statefulset을 정의해주면 된다.
당연히 service selector와 동일한 selector를 spec에 넣어줘야 한다.

apiVersion: apps/v1
kind: StatefulSet

metadata:
  name: mysql

spec:
  serviceName: mysql
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8
          env:
            - name: MYSQL_ROOT_PASSWORD
              value: password
          ports:
            - containerPort: 3306
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 10Gi

여기서 중요한 것 중 하나는 volumeClaimTemplates다.

이게 PVC에 대한 요청값이다.
이걸 적절히 설정해놓으면, PV 관리자가 받아서 Pod마다 하나씩 할당해준다.

그리고 한번 실행해보자.

그러면 실제로 Pod들이

0번


1번


2번 순으로 천천히 생성될 것이다.
statefulset은 이전 Pod가 완성되어야 그 다음 Pod를 실행하는 느긋한 구조를 가진다.
더 정확히 말하면 podManagementPolicy가 OrderedReady가 될때까지 기다린다.

그리고 만약 삭제되거나 replica 개수가 축소된다면, 그 역순으로 제거해나간다.

pvc도 Pod마다 하나씩 할당이 되어있을 것이다.

내 경우에는 k3s라서 local-path로 자동으로 pvc가 할당되었는데, 이건 pv 구성에 따라 달라진다.
