[Concurrency] thread local storage (TLS)

  • 2026.06.23
  • ::
  • 3 min read
#동시성
[원본 링크]

thread local storage는 스레드마다 독립적인 전역변수를 할당하는 사용 패턴을 말한다.
thread local이라고 부르기도 하고, TLS라고도 부른다. 더 유명한 축약어가 있어서 헷갈리긴 한다만.

아무튼 동시성 기능을 제공하는 주요 언어들에는 거의 다 있다.
C/C++, Rust, Java, C#, Python 등에서 기본적으로 제공되고, Go가 특이하게 예외적으로 없다.

사실 이게 문법적으로 지원되지 않아도, 비슷한 의미론은 어떻게든 구현할 수 있다.
어쨌든 스레드의 생명주기 전체에서 살아있고 접근이 되면 되는 것이기 때문이다. 스레드 스코드 최상위에 두고 참조할 수 있게 전달만 하면 비슷하게 동작을 한다.




C/C++에서

C++11에서는 컴파일러 수준에서 thread_local라는 전용 키워드를 제공한다.
이걸 붙인 채로 전역변수로 선언하고, 참조해서 사용하면 자동으로 스레드마다 복제가 된다.

C도 C11, C23에서는 _Thread_local라는 동등한 키워드를 추가했다.

그 이전 버전의 클래식 C에서는 컴파일러별 비표준 확장을 통해서만 사용할 수 있다.




Rust에서

Rust도 기본 라이브러리 수준에서 스레드 로컬을 지원한다.

이러고 사용하면 알아서 스레드마다 복제가 된다.




내부 구현

이게 그냥 전역변수처럼 적당히 변수 선언하고 끝내는 식으로 생각하기 쉽지만, 실제 내부 구현은 다층적인 편이다.
사실, thread local은 단순히 컴파일러의 문법적 설탕이 아니다. OS, CPU 수준 지원을 모두 활용해서 만들어지는 것이다.

먼저 CPU에서 제공하는 적절한 인터페이스가 있다.
x86의 경우에는 fs라는 게 있는데, TLS를 위해서 만들어진 것은 아니지만 가장 사용하기 적절하고 실제로도 거의 ABI 수준에서 약속이 되어있다.
TLS에 대한 접근은 컴파일러에 의해 다음과 같이 변환된다.

mov rax, fs:[offset]

fs라는 전용 공간에서 값을 접근하는 것이다.

그리고 OS에서는 이를 이용해서 OS 스레드별 TLS 매핑을 관리한다.
thread local 변수를 선언하면 내부적으로 TLS Block이란 것을 만든다. 이걸로 OS Thread => fs 매핑을 제어하는 것이다.

그래야 컨텍스트 스위칭이 발생할때마다 fs 값을 적절히 교체할 수 있기 때문이다.




언제 써야하나?

사실 thread local을 사용할만한 상황이 그렇게 자주 나오진 않는다.
일반적인 응용프로그램에서는 적당히 로컬변수만으로 처리할 수 있다. 스레드별로 살아있는 전역 값이 필요하다면 공유 객체를 잘 설계해서 넘기면 되는 것이다.

성능적으로는 좀 애매하다. 전역 Mutex 객체와 비교하면 thread local가 빠르긴 한데... 단순 지역변수를 만들고 넘기는 것과 비교하면 비슷하거나 약간 느릴 수도 있다.

다만 C의 경우에는 문법적 표현력의 한계 때문에 thread local이 유용한 경우가 많다. 클로저 캡쳐도 없고, 상위 스코프의 객체를 선언하고 넘겨서 공유하는 것이 매우 지리하고 불편하기 때문이다.

그리고 이게 스레드별 상태를 고정적으로 할당하고 사용하는 것이기 때문에, 실제 스레드가 동적으로 교체될 수 있는 그린 스레드(혹은 가상 스레드) 개념과는 잘 맞지 않을 수 있다. 게다가 숨겨진 전역 상태가 암시적으로 공유되니, 전역변수와 비슷한 이유에서 코드 품질이 떨어질 수 있다.
이게 Go가 TLS를 배제한 이유이기도 하다. 마찬가지로 Java에서도 Virtual Thread가 도입된 이후로는 도입을 경계하는 분위기가 생긴 것 같더라.
