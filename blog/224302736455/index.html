S3는 편하기도 하지만 사용량이 늘게 되면 상당한 골칫거리가 된다.
사용 패턴을 확인하고 비용을 최적화하는 것이 상당히 까다롭기 때문이다. 도대체 무슨 파일들이 용량을 먹고 있는지, 지워도 되는건지 확인하기가 참 난감하다.

S3 Inventory는 그런 상황에서 파일 목록을 추적하기 위한 수단이다. 어떤 파일들이 어떤 경로로 있는지, 크기는 얼마나 되는지 등을 자동으로 뽑아서 다른 S3 버킷에 보고서로 정리해주는 기능을 제공한다.
LIST API 써서 직접 받아와도 되긴 하는데, 너무 오래걸리고, 비용도 비용대로 들고, 불편하다.

이렇게 말이다.

그럼 저걸 다운받든, athena를 쓰든 해서 읽으면 된다.
가장 간단한건 그냥 athena 붙여서 쿼리를 바로 날리는 것이다.

CREATE EXTERNAL TABLE s3_inventory (
  bucket                STRING,
  key                   STRING,
  size                  BIGINT,
  last_modified_date    STRING,
  storage_class         STRING
)
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.OpenCSVSerde'
STORED AS INPUTFORMAT  'org.apache.hadoop.mapred.TextInputFormat'
         OUTPUTFORMAT  'org.apache.hadoop.hive.ql.io.HiveIgnoreKeyTextOutputFormat'
LOCATION 's3://<dest-bucket>/<dest-prefix>/<src-bucket>/<config-id>/data/'
TBLPROPERTIES ('skip.header.line.count'='0');

그럼 이런 식으로 잘 뽑혀나올 것이다.


SELECT
  split_part(key, '/', 1)                              AS top_dir,
  COUNT(*)                                             AS object_count,
  SUM(CAST(size AS BIGINT))                            AS total_bytes,
  ROUND(AVG(CAST(size AS BIGINT)), 0)                  AS avg_bytes,
  MIN(CAST(size AS BIGINT))                            AS min_bytes,
  MAX(CAST(size AS BIGINT))                            AS max_bytes,
  -- 사람이 보기 좋은 단위(GB/MB)
  ROUND(SUM(CAST(size AS BIGINT)) / 1024.0/1024/1024, 2) AS total_gb,
  ROUND(AVG(CAST(size AS BIGINT)) / 1024.0/1024, 2)      AS avg_mb,
  ROUND(MAX(CAST(size AS BIGINT)) / 1024.0/1024, 2)      AS max_mb
FROM s3_inventory
GROUP BY split_part(key, '/', 1)
ORDER BY total_bytes DESC;

최상위 prefix 기준으로 집계하는 단순한 쿼리다. 이렇게 쫓아가면 된다.
