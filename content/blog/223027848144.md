deployment는 파드를 관리하는 그룹 개념이라고 볼 수 있다.  
정확히는 파드에 대한 컨트롤러(controller)라고 부른다. 컨트롤러는 deployment 말고도 몇가지가 더 있지만, 가장 대표적으로 쓰이는 놈이 이놈이다.  

실제로 많은 리소스가 필요한 경우에는 단일 파드로만은 감당이 불가능한데, 뭐 띄우고 배포할때마다 파드를 직접 다 관리하는건 상당히 번거로운 일이다.  

그래서 보통 서버같은걸 띄운다고 하면 deployment 단위를 구성해서 올리곤 한다.  
deployment는 replica 등의 좀더 발전된 상위 제어 개념이 제공된다.

## deployment 만들어보기

deployment는 create deployment 명령을 사용해서 만들 수 있다.  
한번 이미지 하나 갖다가 만들어보자.  

```javascript
kubectl create deployment test-server --image=myyrakle/node-server-for-test
```
도커허브에 있는 myyrakle/node-server-for-test이미지로 test-server라는 이름의 deployment를 만든다는 뜻이다.  
myyrakle/node-server-for-test 이미지는 내가 급조해서 올린 것인데, 80포트로 심플한 HTTP 서버를 올려주는 것만 딱 들어있다.  
코드나 구성을 보고 싶다면, 아래 레포지토리를 참조해보면 되겠다.  
[https://github.com/myyrakle/infrastructures/tree/master/k8s/examples/conrollers/deployments/node-server-for-test](https://github.com/myyrakle/infrastructures/tree/master/k8s/examples/conrollers/deployments/node-server-for-test)

## deployment 스케일링하기 (replica set)

deployment의 장점 중 하나는, 언급했던대로 파드를 레플리카 단위로 관리할 수 있다는 것이다.  
한번 test-server의 레플리카를 3개로 늘려보겠다.  
이런식으로 하면 된다.  

```javascript
kubectl scale deployment test-server --replicas=3
```

![](/images/img_af22303998bb9349.png)
그럼 이제 이렇게 파드를 3개나 띄워주게 된다.  

또 0으로 하면 줄여서 활성 파드를 없앨 수도 있다.  

![](/images/img_c75058d0f34bca15.png)

## deployment 삭제하기

이번에는 만들어뒀던 deployment를 삭제해보겠다.  
delete deployment 명령을 사용하면 된다.  

```javascript
kubectl delete deployment test-server
```

## yaml로 말아보기

위에서는 create deployments 명령을 사용해서 deployment를 바로 만들어봤지만, 실제로 응용할 때는 그렇게 하지 않고 yaml로 구성을 말아놓고 사용한다.  

위에서 했던걸 yaml로 구성하면 아래와 같은 형태가 된다.  

```javascript
apiVersion: apps/v1
kind: Deployment
metadata:
  name: test-deploy
spec:
  replicas: 3
  selector: 
    matchLabels: 
      app: test-server
  template: 
    metadata: 
      labels: 
        app: test-server
    spec: 
      containers: 
        - name: test-server
          image: myyrakle/node-server-for-test
```
replicas를 3개로 하고, 이미지는 동일하게 가져왔다.  
저렇게 파일을 만들어놓고  

kubectl create 명령으로 파일만 넘겨주면 된다.  

![](/images/img_bbbdd9461eb77a82.png)
그럼 이렇게 잘 만들어질 것이다.

## vs ReplicaSet

deployment와 비슷한 기능으로 replicaset이라는게 있는데, 기본적으로 레플리카 단위로 파드를 관리한다는 것은 동일하다. 하지만 replicaset은 배포에 대한 방법론을 제공하지 않아서 무중단배포를 구성할 수가 없었다.  
이런 배포에 대한 단점을 개선해서 나온 것이 deployment다.  

그래서 deployment는 배포에 대한 몇가지 옵션을 제공한다. 이건 별도 포스트로 정리해보도록 하겠다.
