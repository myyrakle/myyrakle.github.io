DynamoDB는 날강도들 가득한 AWS에서 가장 강력하고 비용 효율적인 DB 서비스다.  
RDB만큼 유연한 검색기능을 제공하진 않지만, 무제한 쓰기 확장이 가능하고, 비용도 성능 대비 저렴한 편이기 때문이다. 그래서 로그 같은 APPEND Only 데이터 수집 창고로 쓰기 좋다.  

그리고 DynamoDB의 또 하나의 장점은, Athena & S3 파이프라인과도 아주 잘 통합된다는 것이다.  
서비스가 DynamoDB에 데이터를 쌓으면, 그걸 S3에 parquet으로 말아서 올리고, Athena로 대량 집계를 날리는 일련의 패턴을 편리하게 구성할 수 있다.

## 아키텍쳐 선택지

Athena에서 DynamoDB를 소스로 사용하는 방법은 여러가지가 있다.  

<br>

#### 1. 직구로 꽂기
가장 단순한 사용 방식은 다음과 같다.  
DynamoDB Connector를 사용해서 직통으로 연결하는 것이다.  

![](/images/img_897e07379d103c2b.png)
이건 성능이나 비용상의 이점이 거의 없다.  
그냥 DynamoDB에 대신 READ 날려서 SQL Query에 맞춰서 가공을 하는 것에 불과하고, DynamoDB의 성능 한계를 그대로 받는다. 비용도 DynamoDB RCU 기반으로 뜯는다.  
이건 작은 데이터에 대해서 JOIN 하는 용도로만 쓰거나 해야한다.  

<br>

<br>

#### 2. S3 Export Only
그 다음으로 나은 방법은 다음과 같다.  
DynamoDB는 PITR 백업 기반으로 S3에 export하는 기능을 제공하는데, 그러고 나면 아테나에서 바로 꽂아서 읽을 수 있다.  

![](/images/img_e346e46b6786ce32.png)
근데 이것도 사실 최선이라고 할 수는 없다.  
아테나 조회 패턴에 최적화되어있지도 않고, 공간도 불필요하게 크게 먹기 때문이다.  
성능과 비용효율성을 원한다면 parquet으로 말아놓는 구성이 필요하지만, 이건 그냥 flat하게 펼쳐버린다.  

<br>

<br>

#### 3. S3 Export & CTAS (parquet)

Athena CTAS라는 기능을 쓰면 Athena만을 위한 최적화된 구조로 재생성시킬 수 있다.  

![](/images/img_4603ac3dc9a15615.png)

<br>

<br>

#### 4. Best Practice?

Best Practice로 간주되는 방법은 다음과 같다.  

![](/images/img_a8e0e3e7b7b06405.png)
증분형 S3 export를 통해서 S3에 붓고, 그걸 Athena를 사용해서 재생성하는 것이다.

## 매핑용 External 테이블 구성하기

자. 이제 준비는 거의 됐다.  
Athena로 이동해서 매핑 테이블을 구성해준다.  
방금 Export된 json.gz들이 있는 경로를 넣고, 어떤 필드들이 있는지 체크해서 키 목록을 세팅해주면 된다.  

```javascript
CREATE EXTERNAL TABLE 외부테이블구성 (
    Item struct<
        키:   struct<S:타입>,
        ...
    >
)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION 's3://버킷/경로/dynamo/AWSDynamoDB/?/data/';
```

![](/images/img_e168fe4ea1b16026.png)
이것 자체는 기존 데이터에 메타데이터만 추가해주는거라서, 비용이나 성능을 먹지는 않는다.  

<br>

![](/images/img_be8da9ff87647095.png)
그럼 일단 조회를 해볼 수는 있을 것이다.  

우린 여기서 한발 더 나아가야 한다.  
athena의 CTAS 기능을 사용해서 parquet로 재생성해준다.  

```javascript
CREATE TABLE access_logs_parquet
WITH (
  format = 'PARQUET',
  parquet_compression = 'SNAPPY',
  external_location = 's3://저장경로/'
) AS
SELECT
  Item.id.S   AS id,
  Item.ip.S   AS ip,
  Item.path.S AS path
FROM 임시테이블;
```

![](/images/img_54c66aaa46b522f3.png)

그럼 실제로 최종 목적지에 압축된 parquet 엔트리가 생성될 것이고  

![](/images/img_20d9320808c5b5f5.png)

쿼리도 날려볼 수 있을 것이다.  

![](/images/img_4ec425c31f309905.png)
잘 들어간 것을 볼 수 있다.  
쿼리 날리는건 그냥 이렇게 하면 된다.

## DynamoDB의 증분 Export (Incremental Export)

근데 방금 사용한 방식은 잘 동작하긴 하지만, 확장성이 좋은 방식은 아니다.  
매번 전체 데이터를 가져와서 밀어넣는 것은 너무 오래걸리고, 비용도 비용대로 먹는다.  

다행히 DynamoDB는 이런 딜레마를 잘 해소해준다.  
DynamoDB의 백업 시스템 자체가 타임스탬프 기반으로 필터링이 되는 구조로 되어있어서, "몇시간 전부터의 데이터"나 "어제부터의 데이터" 같은 식으로 부분적 EXPORT가 잘 되기 때문이다.  

이번에는 행을 하나 추가한 다음에 동기화를 구성해보자.  

![](/images/img_300c75a5d774b4ca.png)
id=4444를 추가하고  

<br>

![](/images/img_59d19ce0dab63fa5.png)
증분 내보내기를 선택해서 트리거한다.  

<br>

![](/images/img_6761ec40424d194d.png)
그럼 마찬가지로 json.gz가 잔뜩 떨어진다.  

근데 여기서부턴 좀 다르다.  
전체 내보내기는 진짜 전체 데이터만 가져오는 거라서 구조가 단순한데, 증분 내보내기는 UPDATE/DELETE 같은 상태 정보도 함께 포함되기 때문이다. 그 시기의 "모든 변경"을 가져온다.  

```javascript
CREATE EXTERNAL TABLE access_log_incremental (
    Metadata struct<
        WriteTimestampMicros: string
    >,
    Keys struct<
        id: struct<S:string>
    >,
    NewImage struct<
        id:   struct<S:string>,
        ip:   struct<S:string>,
        path: struct<S:string>
    >,
    OldImage struct<
        id:   struct<S:string>,
        ip:   struct<S:string>,
        path: struct<S:string>
    >
)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION 's3://버킷/경로/dynamo/AWSDynamoDB/?/data/';
```

![](/images/img_0c1a5d7127cceec2.png)

![](/images/img_ecd1c94984a7da0b.png)
그래서 이런 식으로 뭐가 좀 많다.  
INSERT/UPDATE는 NewImage를 읽어서 넣어야 하고, DELETE는 Keys만 전달된다.  

여기서부터는 기능 구조에 따라서 제약조건이 좀 달라진다.  
APPEND-ONLY의 단순한 시스템이라면, 그냥 NewImage만 걸러서 기존 CTAS 테이블에 insert하면 된다.  

![](/images/img_b4808a169649c9c6.png)

![](/images/img_cad635f9812b2372.png)
그럼 자동으로 parquet 덩어리가 추가될 것이고  

<br>

![](/images/img_c2cecc0d06c68bbd.png)
조회도 잘 될 것이다.  

근데 UPDATE/DELETE를 고려해야 한다면 좀 복잡해진다.  

사실 지금 위의 쿼리도, 기존 parquet에 들어있던 ID와 동일한 INSERT가 들어온다면 중복 ID가 발생할 수 있다는 단점이 있다. 원본 데이터부터가 NEW INSERT ONLY라는 전제 하에서만 잘 동작하는 구조다.

## IceBerg를 사용한 UPDATE/DELETE

여기서 문제가 되는 것은 parquet 자체가 수정을 위한 데이터 규격이 아니기 때문이다.  
그저 데이터를 압축해서 효율적으로 관리하기 위한 포맷에 불과하다.  

그래서 변경되는 데이터셋에 대해 수정과 삭제를 고려해서 parquet을 관리해야 한다면, iceberg 엔진을 사용하는 편이 권장된다.  
다음과 같은 식으로 테이블을 만들면 iceberg의 파티션 관리 능력을 가져갈 수 있다.  

```javascript
CREATE TABLE access_log_iceberg
WITH (
  table_type = 'ICEBERG',
  format = 'PARQUET',
  location = 's3://.../iceberg/access_log/',
  is_external = false
) AS
SELECT id, ip, path FROM access_log_parquet;
```

그러면 이제 MERGE 구문을 통해서 파티션을 재구성하는 것이 가능해진다.  
DELETE나 UPDATE를 반영할 수 있다.  

```javascript
MERGE INTO access_log_iceberg t
USING (
  SELECT
    COALESCE(NewImage.id.S, Keys.id.S) AS id,
    NewImage.ip.S   AS ip,
    NewImage.path.S AS path,
    NewImage IS NULL AS is_deleted,
    CAST(Metadata.WriteTimestampMicros AS bigint) AS change_ts,
    ROW_NUMBER() OVER (
      PARTITION BY COALESCE(NewImage.id.S, Keys.id.S)
      ORDER BY CAST(Metadata.WriteTimestampMicros AS bigint) DESC
    ) AS rn
  FROM access_log_incremental
) s
ON t.id = s.id AND s.rn = 1
WHEN MATCHED AND s.is_deleted THEN DELETE
WHEN MATCHED THEN UPDATE SET ip = s.ip, path = s.path
WHEN NOT MATCHED AND NOT s.is_deleted THEN INSERT (id, ip, path) VALUES (s.id, s.ip, s.path);
```
