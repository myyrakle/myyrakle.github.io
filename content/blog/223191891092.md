atlas에서는 Elasticsearch와 유사한 구조와 경험을 가지는 search 전용 시스템을 제공한다.  

Elasticsearch처럼 Lucene을 기반으로 형태소 분석을 처리하며, 동의어(synonym), score 기반의 검색 기능 등을 제공한다. 그리고 범용 DB에서 감당하기 힘든 복잡한 필터나 정렬에 대해서도 준수한 성능을 보장한다.  

mongoDB 데이터를 기반으로 바로 생성해서 사용할 수 있다는게 큰 장점이다. 귀찮게 로그스태시 띄워서 동기화 시키고 그러지 않아도 된다.

## 구조

Atlas search는 이런 구조를 가진다.  

![](/images/img_a02ca6e0adf158e7.png)

뒷단에 mongot를 비롯한 백그라운드 시스템을 추가하고, 이를 통해 데이터가 변경될때마다 search 인덱스에 동기화를 시켜준다.  
그리고 유저는 search 인덱스를 통해 값을 검색해서 사용한다.

## 검색해보기

기존에 text 인덱스만을 쓸때는 직접 인덱스를 만들고, $text 등의 필터 절을 통해 필터링을 수행했다.  

![](/images/img_da256ac6436b78a3.png)
[https://blog.naver.com/sssang97/222987710006](https://blog.naver.com/sssang97/222987710006)  
하지만 atlas search에서는 $search라는 전용 필드를 통해 필터링을 수행한다.  

만들어진 search index 탭에서 바로 쿼리를 날려볼 수 있는데  

![](/images/img_1479ca6b31c7ad90.png)
저 쿼리는 name 필드가 "불닭볶음면"인 필드만 검색해서 반환하는 간단한 쿼리다.  

그리고 기존 mongoDB api를 통해서도 사용할 수가 있다. aggregate 함수에 대해서 적용할 수가 있는데  

![](/images/img_75403df8e0e4f576.png)
이렇게 $search라고 atlas search만을 위해 예약된 필드가 있다.  
아마 altas 밖에서는 유효하지 않을 것이다.  

검색에 대한 더 상세한 내용은 별도 포스트로 정리해보겠다.

## 인덱스 매핑

별다른 설정을 주지 않고 인덱스를 만들면 이런식으로 구성이 될 것이다.  

![](/images/img_8e17e8c3216076e4.png)
이러면 알아서 모든 필드에 대해서 인덱싱을 해주는데, 성능적으로는 비효율적인 부분이 크다.  
디스크도 훨씬 많이 먹고, 인덱싱 성능도 떨어진다고 한다.  

그래서 안정적으로 쓰려면 이렇게, dynamic 매핑을 끄고 정적으로 매핑해야 한다.  

![](/images/img_5100ad6e9e42d8df.png)
