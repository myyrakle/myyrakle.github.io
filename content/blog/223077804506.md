12 Factors는 자체 클라우드 구축과 사용에 있어서 신경써서 관리해야할 12개의 요소, 계층을 정리한 것을 말한다.  

대충 이런 것들이 있다.  

![](/images/img_2b740f7cf608d1cf.png)

<br>

<br>

## 1. codebase

소스코드의 관리를 말한다.  
git이나 svn 같은 프로토콜을 사용하고 remote server를 통해 형상관리를 하는데, 대부분은 git을 사용한다.  
그리고 remote server는 github 같은 서드파티 호스팅을 쓸 수도 있고, gitlab 등으로 자체 git 서버를 구축할 수도 있다.  

권고사항  
1. 하나의 앱은 하나의 코드베이스를 가져야 한다.  

<br>

<br>

<br>

## 2. dependencies

애플리케이션의 디펜던시를 명확하게 관리하라는 말이다.  
자바면 gradle이나 maven을 사용할 것이고, Nodejs면 npm 등을 사용할 것이다.  

어지간히 원시적인 언어 환경이 아니라면 자연스레 갖추게 되는 부분이다.  

권고사항  
1. 디펜던시는 최대한 명확하게. latest 같은건 자제  

<br>

<br>

<br>

## 3. config

이건 애플리케이션의 환경변수라고 이해하면 편하다.  

데이터베이스 접속정보나 인증용 키 등의 구성정보를 코드와 분리해서 관리하라는 것이다.  

스프링의 경우 Spring Cloud Config를 사용하곤 한다.  

<br>

<br>

<br>

## 4. backing services

Backing Service는 네트워크를 통해서 사용할 수 있는 모든 서비스를 말한다.  

마이크로서비스 단위라고 할 수도 있다.  

필요에 따라 분리하고 연동하면 된다.  

<br>

<br>

<br>

## 5. build, release, run

흔히 말하는 CI/CD를 말한다.  

소스코드를 빌드하고, 환경에 릴리즈하고, 릴리즈한걸 실행하는 3개의 구분을 명확히 하란 것이다.  

<br>

<br>

<br>

## 6. process

각 애플리케이션은 상태가 없는(stateless) 프로세스를 지향해야 한다.  

그래서 세션 같은 메모리 데이터를 지속적으로 관리하려면 중앙화된 Redis를 쓰거나 한다.  

<br>

<br>

<br>

## 7. port binding

애플리케이션의 포트를 열어서 다른 애플리케이션이 접속할 수 있도록 하자는 것이다.  
내부 애플리케이션도 이를 위해 백엔드 서비스처럼 URL을 제공해야 한다.  

스프링에서 이와 관련된 라이브러리로 Spring Cloud Netflix가 있다.  

<br>

<br>

<br>

## 8. concurrency (동시성)

애플리케이션을 멀티프로세스로 띄워 동시성 처리를 할 수 있어야 한다.  

사실 Spring 같은 유명한 프레임워크들은 대부분 내부 시스템이 멀티프로세싱을 알아서 처리해준다.  
처리해주지 않는것만 직접 신경써서 구성하면 된다.  

<br>

<br>

<br>

## 9. disposability (처분가능성)

애플리케이션을 빠르게 죽이고 띄울 수 있어야 한다.  

이를 통해 Scale up/down을 더욱 수월하게 달성한다.  

<br>

<br>

<br>

## 10. dev/prod parity

개발용 환경과 서비스 환경을 분리하고 구조적으로 최대한 같게 유지한다.  

이를 통해 테스트-배포 프로세스를 원활히 할 수 있다.  

<br>

<br>

<br>

## 11. logs

애플리케이션의 로그를 각 노드 로컬에 저장하지 말고, 중앙화된 시스템에서 저장-관리한다.  

이를 통해 각 노드에 문제가 생기더라도 로그를 통한 트러블슈팅을 한곳에서 쉽게 할 수 있으며, 로그파일 누적으로 인한 스토리지 초과 문제도 해결할 수 있다.  

<br>

<br>

<br>

## 12. admin process

데이터베이스 초기화나 마이그레이션 등의 일회성 작업을 말한다.  

보통 파이썬 같은 인터프리팅 언어로 만들어 쓰는 편이다.
