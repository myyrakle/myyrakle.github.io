MongoDB도 여타 DB들과 마찬가지로 동시성 제어를 위한 Lock 기능을 제공한다.  

<br>

<br>

## Lock mode

MongoDB는 크게 2가지의 Lock 매커니즘을 지원한다.  

<br>

1. Shared Lock: lock을 점유한 데이터는 다른 write가 들어올 수 없다. 하지만 여러개의 read는 허용한다.(RWLock)  

2. Exclusive Lock: lock을 점유한 데이터는 다른 read/write가 전부 들어올 수 없다.  

<br>

<br>

세분화하면 Intent.. 어쩌고 하는게 더 있긴 한데, 기본적인 원리는 여기에서 벗어나지 않는다.  

그리고 각 연산의 기본 Lock 동작은 다음과 같다. r이 Shared Lock이고, w가 Exclusive Lock이다.  

![](/images/img_d251b6a4b8953a02.png)
[https://www.mongodb.com/docs/manual/faq/concurrency/](https://www.mongodb.com/docs/manual/faq/concurrency/)  
대부분의 mutable 연산이 Exclusive Lock이고 readonly 연산이 Shared Lock임을 알 수 있다.  

이러한 Lock 기능을 통해 MongoDB는 document 단위의 atomic을 보장한다.  

<br>

<br>

<br>

## Lock 충돌시의 동작

동시에 여러개의 operation들을 수행하다보면, Lock이 이미 걸린 상태에서 또다른 write operation이 들어올 경우가 많을 것이다.  
이럴 때 기존의 RDB들은 Lock이 풀릴때까지 기다렸다가 Lock이 풀리면 마저 처리를 하는 방식을 취하는데, MongoDB의 해결방식은 꽤 독특하다.  

![](/images/img_60d7e234098d5948.png)
[https://rastalion.me/mongodb-transaction-management/](https://rastalion.me/mongodb-transaction-management/)  
이미 Lock이 걸려있는 데이터에 write를 시도하면, 몽고 서버는 그에 대해 에러를 던진다.  
그리고 에러가 나면 다시 시도하고, 다시 시도하고... 의 방법을 계속 사용한다.  

MongoDB 내부에서만 일어나는 일이라 사용에 큰 불편함은 없지만, 이게 너무 늘어나면 성능에 지장을 주게 된다.  
CPU 사용량도 매우 커지고, 내부의 WiredTiger 캐시에도 부담을 줘서 전체적인 성능을 저하시키기 때문이다...

## multi-document transaction

이게 뭐냐면, RDB에서 쓰던 그 수동 트랜잭션 기능을 말한다.  
여러개의 operation을 하나로 묶어서 하나의 Lock 단위로 처리하는 것이다.  

MongoDB는 NoSQL이면서도 RDB의 영역까지 노리는 애매한 녀석이라서, 이런것도 비슷하게 지원한다...  

리소스를 대단히 많이 먹을 수 있는 기능이라 권장되지는 않는다.  

사전 조건이 하나 있다. MongoDB 서버가 무조건 단일 인스턴스가 아니라 replicaset 모드여야 한다는 것이다.  
근데 레플리카인거랑 트랜잭션 작동여부가 대체 무슨 상관이지...? 이건 몽고 개발자들 실수인거같다.  

아래는 Node.js 기반의 간단한 transaction  사용 예제다.  

```javascript
const { MongoClient } = require("mongodb");

const url =
  "mongodb://127.0.0.1:27017/?directConnection=true&serverSelectionTimeoutMS=2000&appName=mongosh+1.6.0&replicaSet=rs0";
const client = new MongoClient(url);

async function main() {
  await client.connect();

  // 세션 시작
  const session = client.startSession();

  // 트랜잭션 옵션 설정
  const transactionOptions = {
    readPreference: "primary",
    readConcern: { level: "local" },
    writeConcern: { w: "majority" },
  };

  try {
    await session.withTransaction(async () => {
      const coll1 = client.db("mydb1").collection("foo");
      const coll2 = client.db("mydb2").collection("bar");
      // Important:: You must pass the session to the operations
      await coll1.insertOne({ abc: 1 }, { session });
      await coll2.insertOne({ xyz: 999 }, { session });
    }, transactionOptions);
  } finally {
    // 세션 종료
    await session.endSession();
  }
}

main();
```
