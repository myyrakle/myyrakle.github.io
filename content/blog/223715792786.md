Glue는 AWS에서 제공하는 관리형 데이터 파이프라인 관리 시스템이다.

## 비용

Glue 플랫폼에는 여러가지 기능 요소가 제공되지만, 가장 주된 요소는 데이터 중간 처리 목적의 ETL 기능이다.  
[https://aws.amazon.com/glue/pricing/](https://aws.amazon.com/glue/pricing/)  

ETL의 경우를 들자면, 프로비저닝 비용이 따로 없고 작업에 소모된 시간당*노드당 비용을 청구한다.  

여기서 컴퓨팅 비용은 DPU를 기준으로 측정되는데, 하나의 DPU는 4 vcpu + 16 gb짜리의 고성능 머신이다. DPU 갯수는 작업 크기에 따라서 자동으로 선택된다.  

![](/images/img_d12b3a8e0f1e1a38.png)
실행 단위가 많고 비용도 다 따로라 뭐라고 축약하기 어려운데, 실행시간 기준으로 DPU 1개-1시간당 0.44달러 정도를 부과한다고 보면 된다.

## Source - DynamoDB

먼저 테이블과 데이터를 적당히 준비해놨다.  

![](/images/img_8200d5bd39ebbe61.png)
8개다.  

<br>

![](/images/img_27c49d56b55632ad.png)
테이블에는 PITR이 켜져있어야 한다.  

AWS Glue에서 DynamoDB Source를 사용하는 방법은, 그리 단순하지만은 않다.  

1. 권장하는 방법은 DynamoDB 자체의 S3 export 기능을 활용하는 것이다. 그러면 DynamoDB의 데이터를 S3로 옮긴 다음에 그걸 다시 읽어서 Source로 사용하는 형태가 된다.  
이러면 당연히 추가 비용이 발생한다. 자세한 것은 별도 포스트를 참조한다.  
[https://blog.naver.com/sssang97/223715737789](https://blog.naver.com/sssang97/223715737789)  

2. 두번째 방법은 DynamoDB에서 데이터를 직접 크롤링해서 Source로 사용하는 것이다. 자체적으로 제공하는 크롤링 매커니즘이 있는데, 당연히 DynamDB 읽기 요금 단위가 부과된다.  

나는 지금 1번째 방법을 사용하려 한다.  

DynamoDB를 드래그해서 만들고  

![](/images/img_7100e9baea282242.png)
테이블을 지정한다.  
그리고 S3의 중간 저장 경로를 선택한다.  

그러면 저기다가 DynamoDB 데이터를 옮긴 다음에, 그것을 사용할 것이다.  

버킷은 결과물용으로 따로 만들어놨다.  

![](/images/img_1dfceb5f5763cd41.png)

## 데이터 파티션 - 개수 파티션

파일 하나에 들어가는 행 개수를 조절해보자.  
먼저 2개씩 묶는게 아니라 5개씩 묶도록 파티션 단위 수정을 해보겠다.  

![](/images/img_65b9ba087a0323bf.png)
기존에 이런 코드에다가, 중간에 transform 동작을 쑤셔넣으면 된다.  

<br>

```javascript
from awsglue.dynamicframe import DynamicFrame
```
import하나 추가해주고  

<br>

```javascript
# DynamicFrame을 DataFrame으로 변환
df = AmazonDynamoDB_node1736078338707.toDF()

# 파티션 수 조정 (4개 레코드당 하나의 파일로 묶기)
num_partitions = df.count() // 4
df_repartitioned = df.repartition(num_partitions)

# DataFrame을 다시 DynamicFrame으로 변환
dyf_repartitioned = DynamicFrame.fromDF(df_repartitioned, glueContext, "dyf_repartitioned")
```
이런 식으로 데이터프레임 연산 적당히 거치고, 다시 기존 dynamic frame으로 변환해서 넘겨주면 된다.  

그리고 돌려보면 이번에는  

![](/images/img_a5592d3164fa2ac8.png)
4개씩 2개의 파일로 완성이 될 것이다.
