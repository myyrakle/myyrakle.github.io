Qdrant는 GPU를 통해 검색 속도를 가속하는 기능을 제공한다.  

벡터-벡터간 비교가 핵심 연산이다보니, GPU를 활용한 병렬처리에서 이점을 많이 얻을 수 있는 것이다.  

<br>

<br>

<br>

## 제한사항

1.13버전부터 사용 가능하며, 현 시점에서는 다음과 같은 제한사항이 있다.  

- Qdrant Cloud에서 사용 불가능 (향후 지원 예정)  
- Linux amd64 아키텍쳐에서만 사용 가능. (arm64 미지원)  
- GPU는 nvidia와 amd만 사용 가능.  

이외에도 세그먼트 크기 제한 등이 존재한다.  
참조 [https://qdrant.tech/documentation/guides/running-with-gpu/#known-limitations](https://qdrant.tech/documentation/guides/running-with-gpu/#known-limitations)  

<br>

<br>

<br>

## 사양과 성능

굳이 고성능 GPU를 쓸 필요는 없다.  
여기서 쓰는 병렬연산이라고 해봐야 그렇게 헤비한게 아니라서, 적당히 엔트리급의 GPU를 써도 충분하다.  

그리고 벡터의 길이가 길수록 성능 향상 효과를 더 잘 느낄 수 있다. 벡터/벡터 비교를 병렬화하는 것이기 때문이다.  

공개한 벤치마크에 따르면 2-3배 이상도 빨라질 수 있다고 한다.  
근데 이건 벡터의 크기에 따라서 많이 달라질 것 같다. 벡터의 크기가 크지 않다면 그냥 CPU만 쓰는 것과 성능 차이가 별로 없을 수 있다.

## GPU를 활성화해보기

당연히 GPU와 그 드라이버 세팅은 되어있어야 한다.  
내 경우에는 엔비디아 2080이다.  

![](/images/img_d5068fb510f7ccd6.png)

일단 이 기능은 기존 서버로는 사용할 수도 없다.  
소스에서 빌드한다면 별도 옵션을 줘야 하고, docker도 gpu 버전 이미지가 따로 있다.  

![](/images/img_16a34d4d92bb5849.png)
이 gpu 버전을 받아서 실행하면 된다.  

그냥 docker로 실행하려면 이렇게 하고  

```javascript
docker run \
	--rm \
	--gpus=all \
	-p 16333:6333 \
	-p 16334:6334 \
	-e QDRANT__GPU__INDEXING=1 \
	qdrant/qdrant:v1.15-gpu-nvidia
```

docker compose 예시는 다음과 같다.  

```javascript
  qdrant-gpu:
    image: qdrant/qdrant:v1.15-gpu-nvidia
    container_name: qdrant-gpu
    ports:
      - "16333:6333"
      - "16334:6334"
    volumes:
      - qdrant_gpu:/qdrant/storage
    environment:
      - QDRANT__GPU__INDEXING=1
    ulimits:
      nofile: 65535
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
        limits:
          cpus: "4"
          memory: 16192M
```
당연히 사용할 GPU에 대한 드라이버 설정, 드라이버-컨테이너 설치는 사전에 되어있어야 한다.  

그렇게 실행해서  

![](/images/img_83c6b2605c7de74e.png)
위와 같이 GPU 관련 실행 로그가 뜨면 적용은 잘 된 것이다.

## 성능 벤치마크

대충 256과 512 벡터를 1000만개씩 넣고 때리면서 비교해봤다.  
벤치마크에 사용한 코드는 깃헙에 있다.  
[https://github.com/myyrakle/benchmarks/tree/master/qdrant_benchmark](https://github.com/myyrakle/benchmarks/tree/master/qdrant_benchmark)  

이건 GPU 사용 전 INSERT 속도  

![](/images/img_68a93c64f69574ad.png)

![](/images/img_84261b370e0fe0bb.png)

GPU 사용 전 READ 처리량 - 동시 쿼리 20개  

![](/images/img_8a302778b3fb5f98.png)
256은 초당 235개 정도의 쿼리를 처리하고, 레이턴시는 100ms 정도 됐다.  
512는 초당 92개 정도밖에 처리하지 못했고, 레이턴시는 200ms. 실패도 좀 발생했다. 벡터 크기가 커질수록 성능 안정성이 많이 떨어지는 것 같다.  

GPU 사용 전 READ 처리량 - 동시 쿼리 100개  

![](/images/img_70923ac6724ef5c2.png)
동시 쿼리를 늘린다고 눈에 띄게 처리량이 늘진 않았다. 512의 경우에는 실패율이 생겼다.  

그럼 GPU 모드로 찌르면 어떨까? 경천동지할 성능 향상이 있을까?  
동시 쿼리 20개 정도만 밀어넣어봤다.  

![](/images/img_eb1db78f86d43d03.png)
일단 256의 경우에는 성능 향상이 있긴 한데 미미하다.  
TPS가 235 => 284니까 대충 20% 정도는 향상이 된 것이다.  

<br>

![](/images/img_a56a9af6c215da9e.png)

![](/images/img_e29e4f0e46815ca9.png)
근데 512는 좀 이상했다.  
처리도 엄청 느리고, 동시 처리량도 블락이 걸린것처럼 다 막히고 뻗었다.  

뭔가 버그가 좀 있는 것 같았다.  
똑같은 조건으로 256과 512를 전부 초기화했는데도 한쪽에만 이런 증상이 나왔다.  

아직 안정화되었다고 보기는 이른듯...
