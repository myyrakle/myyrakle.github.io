Athena는 S3에 대한 SQL 쿼리 기능을 제공하는 서비스다.  

대형 데이터같은걸 다루다보면 S3 같은 스토리지에 파일을 기록하게 될 일이 꽤 많다.  

근데 S3에 저장하는건 그렇다 쳐도, 꺼내쓰는 것은 상황이나 취향에 따라 여러가지 선택지가 있다.  

대표적으로 AWS의 redshift나 GCP의 bigquery 등이 이런 빅데이터 분석에 최적화된 도구다. 이러한 도구들의 단점은, 아무래도 동기화를 추가로 시켜줘야 하고, 그런데서 발생하는 전송 비용이나 번거로움이 존재한다는 것이다.  

반면 Athena는 별다른 처리 없이 S3에 바로 액세스할 수 있게 해주는 기능을 제공한다.  
Presto, Apache Spark, Aparch Hive 등을 기반으로 동작하며, 페타바이트 단위의 데이터도 처리 가능하다.

## vs Redshift

빅데이터 분석을 위한 기능이라는 점에서는 Redshift와 장르가 비슷하지만, 영역이 완전히 겹치지는 않는다.  

Redshift는 복잡한 구조를 가진 데이터셋 처리에 능하다. 여러가지 데이터를 구겨넣고 빠른 성능으로 대규모 JOIN 작업도 무리없이 해낼 수 있다.  
반면 Athena는 단일 데이터셋에 대한 처리에 최적화되어있으며, JOIN 같은 작업에는 성능한계가 있다.  

사용성에서는 Athena가 큰 강점을 가진다. Redshift의 경우에는 별도의 프로그램을 구성해서 동기화를 해줘야 하지만, Athena는 몇번의 클릭만으로 S3를 통째로 바로 로드할 수 있다.  

비용 면에서는 논란이 있을 수 있다. 하지만 Redshift는 비용 계산이 매우 복잡한 반면에 Athena는 꽤나 간단하고 명료한 과금 체계를 가지고 있다.  

성능은 마치 RDB vs NoSQL의 대결과 비슷하다.  
단순 Read나 Aggregate는 대체로 Athena가 약간 더 빠르다.  
하지만 테이블간 Join에 대해서는 Redshift가 확실하게 더 빠르다.

## 테이블 만들기

테이블 생성 쿼리는 마법사를 써서 짜는게 편하다.  

![](/images/img_ad3fb94fdfdb7957.png)

이름 짓고 데이터베이스 고르고  

![](/images/img_234a4cdecc703a91.png)

테스트데이터가 들어있는 경로를 넣어준다.  

![](/images/img_b00c9ae35d3f262a.png)
경로는 /로 끝나야하고, 아테나는 저기 들어있는 파일을 전부 풀스캔때린다.  

그리고 입력 파일 형태도 정해준다.  

![](/images/img_9453c7f624b700f0.png)
내가 csv로 올렸으니까, csv로 하면 된다.  

그리고 컬럼을 순서대로 정의한다.  

![](/images/img_ea86a251b5db54b5.png)

그럼 이러한 형태로 쿼리가 생성될 것이다.  

![](/images/img_f0de3fdf6da5be60.png)
생성해준다.  

<br>

![](/images/img_2b6c01ca22925ebf.png)
그래서 이렇게 왼쪽에 테이블이 생성되면 잘 된 것이다.  

csv만 처리할거라면 이렇게 하는게 좋다.  

```javascript
CREATE EXTERNAL TABLE IF NOT EXISTS `DB명`.`테이블명` (
  ...
)
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.OpenCSVSerde' 
WITH SERDEPROPERTIES ("separatorChar" = ",", "escapeChar" = "\\") 
LOCATION 's3://...-cost/aws/디렉터리/' -- 읽어올 csv 파일이 존재하는 디렉터리
TBLPROPERTIES ("skip.header.line.count"="1") -- 첫번째 row는 생략
```
