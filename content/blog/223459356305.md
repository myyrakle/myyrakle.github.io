podman은 Redhat에서 만들고 관리중에 있는 컨테이너 스택이다. 2018년 즈음부터 시작됐다.  

docker의 경쟁자다.  

<br>

<br>

<br>

## vs Docker

podman은 runc 기반으로 자체적인 컨테이너 시스템을 구현한다. Docker의 역할을 대체할 수 있다는 소리다.  
그리고 기본적인 구조에 있어서도 차이가 많다.  

예를 들어 docker 시스템은 daemon을 통해 루트권한으로 실행되는데, 이런건 사실 보안적으로 취약점이 될 소지가 많다. podman의 경우에는 사용자 단위로 컨테이너를 띄우기 때문에 보안적 측면에서의 이점이 있다.  

Docker보다 가볍고 빠르다고 하는데... 명확하게 검증된건 없다. 마케팅이다.  

의도적으로 사용 방식 자체는 Docker와 거의 동일하게 맞춰놨다. 대부분의 Docker CLI 명령 형식은 podman CLI에서도 사용할 수 있다.  

Docker와의 가장 큰 차이점은 "Pod" 단위를 지원한다는 것이다. k8s에서 사용하는 pod와 동일한 개념이다.  
pod를 통해 여러개의 컨테이너를 하나의 단위로 묶을 수 있다.  

podman은 솔직히 주변 환경이나 자료 수준에 있어서 Docker의 발끝에도 미치지 못한다.  

아 그리고, k8s가 Docker는 아니지만 Docker image 규격을 사용하듯이, podman도 그렇다.  

<br>

<br>

<br>

## with k8s

쿠버네티스와 사용사례가 상충되지는 않는다. 보완적인 관계에 있다.  

어차피 고-급 엔터프라이즈 레벨로 가면 Docker나 Podman을 서비스 수준에서 그냥 쓰지는 않을 것이고, 결국에는 k8s 스택으로 갈 것이다.  

그러면 docker나 podman이나 그냥 개발환경에서 서버 띄우는 정도로만 응용이 될 것이다. 편한걸로 고르면 된다.  

다만 podman은 pod를 k8s yaml로 변환하거나, k8s yaml을 import하는 상호호환 기능이 좀 있어서 더 편할 수 있다.

## 설치

데비안 계열의 경우에는 그냥 기본 패키지매니저를 통해서 바로 설치할 수 있다.  

```javascript
sudo apt-get -y install podman
```

![](/images/img_48027b42b4e273ed.png)

다른 환경에서의 설치 방법들도 문서에 잘 정리되어있다.  
[https://podman.io/docs/installation](https://podman.io/docs/installation)  

아무튼 설치만 마치면 podman 데몬이 자동으로 떠있을 것이다.

cli를 통해서 바로 제어해볼 수 있다.

## 기본 사용법

조금씩 다른 부분이 있긴 하지만, 대략적인 사용 방식은 Docker와 크게 다르지 않다.  

![](/images/img_f007b18d95f22146.png)
익숙한 명령어셋일 것이다.  

예를 들어 도커 이미지를 받아오려면 pull 명령으로 받을 수 있다.  

![](/images/img_2941b41fb23e5c19.png)

![](/images/img_cd29c0f1db57fe01.png)
잘 받아진 것을 볼 수 있다.  
차이점이라면, docker에서 레지스트리 경로를 적지 않아도 docker 기본 레지스트리로 연결해주던 것과 다르게, 직접 다 명시해줘야 한다는 것 정도가 있겠다.  

실행법에도 별 차이는 없다.  
아래 명령은 httpd(apache) 서버 컨테이너를 8080포트로 띄우는 간단한 명령이다.  

```javascript
sudo podman run -dt -p 8080:80/tcp docker.io/library/httpd
```
