tokio는 std::sync의 Mutex와 별개로 자체적인 Mutex 구현체를 제공한다.  

대체로는 같은 결과를 얻을 수 있지만, 동작 방식이 조금 다름에 유의해야 한다. 일단은, tokio 문서에서도 가급적 std의 Mutex를 사용하는 것을 권장하는 편이다.  

둘다 섞어쓰는건 문제가 없다.  

<br>

<br>

<br>

## 사용법

일단 사용법은 표준 뮤텍스와 크게 다르지는 않다.  

![](/images/img_b4d2846e977aeedd.png)
생성방식은 동일한데  

<br>

![](/images/img_c792ddd55ae25f7c.png)
뮤텍스 가드 획득에 Result 레이어가 없고 await을 통해 대기한다는 것이 좀 다르다.  

아래는 간단한 예제 코드 전체다.  

```javascript
use std::sync::Arc;

use tokio::sync::Mutex;

#[tokio::main]
async fn main() {
    let _counter = Arc::new(Mutex::new(0));

    let counter = _counter.clone();
    let task1 = tokio::spawn(async move {
        for _ in 0..1000 {
            let mut counter = counter.lock().await;
            *counter += 1;
        }
    });

    let counter = _counter.clone();
    let task2 = tokio::spawn(async move {
        for _ in 0..1000 {
            let mut counter = counter.lock().await;
            *counter += 1;
        }
    });

    let (_1, _2) = tokio::join!(task1, task2);
    let counter = _counter.lock().await;
    println!("Final counter value: {}", *counter);
}
```

## 뭐가 다른가?

표준 Mutex는 lock을 얻을 수 없는 상태면 일단 현재 컨텍스트를 차단한다. 물론 OS 기능들을 활용하기 때문에 비효율적인 차단을 하지는 않는다.  
tokio Mutex는 lock을 얻을 수 없는 상태면 tokio executor에게 실행을 양보한다. FIFO 기반으로 우선순위가 지정된다.  
os 수준에서 block 리소스를 아끼는지, tokio 수준에서 아끼는지의 차이일 뿐이다.  

근데 비동기 컨텍스트에서라도 표준 Mutex가 비용이 더 적고 효율적인 경우가 많다.  
특히 Mutex 내부 값이 단순한 데이터성 값이라면 그냥 표준 Mutex가 훨씬 적합하다.  

tokio Mutex가 적합한 경우는 데이터베이스 연결과 같은 I/O 리소스를 다룰 때다. 그런 특별한 케이스가 아니라면 표준 Mutex를 쓰는 것을 tokio에서도 권한다.
