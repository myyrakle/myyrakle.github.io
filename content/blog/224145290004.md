Qdrant는 텍스트 검색에 대한 기능도 꽤 잘 제공한다.  
기능의 범위가 Elasticsearch보다는 못하지만, 가벼운 사용사례에서는 쓸만하다.  

qdrant의 Full Text Filtering은 Payload Index의 한가지 변종으로서 지원된다.  

<br>

<br>

<br>

## 단어 분리 (Tokenizer)

Qdrant는 Elasticsearch와 마찬가지로, 텍스트를 형태소 기반으로 나눠서 inverted index 구조 기반으로 빠르고 품질 높은 검색을 수행할 수 있다.  

![](/images/img_04cd3d10b5a51efb.png)

이를 위해 지원되는 tokenizer 목록은 다음과 같다.  

whitespace - 공백 기반 분리  
word - 공백 및 구두점, 특수문자 기반 분리  
prefix - word와 동일한 조건 하에 접두사 검색용 분리 (red => r, re, red)  
multilingual - 자연어의 문장 구성요소를 파악해서 형태소 분석 (다국어 가능)  

위에서 아래로 갈수록 섬세한 토큰화가 가능하나 그만큼 오버헤드가 증가한다.  
단순한 사용사례라면 word 정도로 충분할테지만, 한국어에 대한 자연어 분석 같은걸 해야한다면 multilingual을 사용해야 한다.  

multilingual는 내부적으로 meilisearch 팀에서 만든 형태소 분석기를 사용한다.  
이건 현재  라틴문자, 중국어, 한국어, 태국어, 일본어, 아랍어 등을 지원한다.  
지원 목록에 대한 상세 내용은 아래 페이지를 참조하면 된다.  
[https://github.com/meilisearch/charabia](https://github.com/meilisearch/charabia) (일본어 제외)  
[https://github.com/daac-tools/vaporetto](https://github.com/daac-tools/vaporetto) (일본어 한정)

## 기본 사용법

텍스트 검색을 쓰려면 인덱스를 먼저 만들어야 한다.  

다른 기본 필터링들은 인덱스가 없어도 성능이 느릴뿐 동작은 하지만, text 검색은 전용 text 인덱스가 반드시 존재해야 한다. 생성 방식 자체는 다른 payload index들과 비슷하나, 옵션이 매우 많다는게 차이점이다.  

```javascript
PUT /collections/컬렉션명/index
{
    "field_name": "필드명",
    "field_schema": {
        "type": "text",
        "tokenizer": "word"
    }
}
```

![](/images/img_568d52e5b666031b.png)

<br>

<br>

#### text 검색 (문장 단어 무조건 포함)

만들고 나면, 이제 filter 등을 통해서 텍스트 검색을 수행할 수 있게 될 것이다.  
사용법 자체는 단순하다. 그냥 text 절을 사용하면 된다.  

```javascript
POST /collections/testdb/points/query
{
  "with_payload": true,
  "filter": {
    "must": [
      {
        "key": "name",
        "match": {
          "text": "red"
        }
      }
    ]
  }
}
```

![](/images/img_f5e6080096c29ced.png)
그러면 위와 같이 해당 token(red)가 포함된 것들을 필터링해줄 것이다.  

<br>

![](/images/img_585166861ee8ae0e.png)
단, text 필터는 단어가 겹치는 모든 것을 가져오진 않는다. 현재 검색어에 포함된 모든 단어가 포함되는 항목을 가져오기 때문에, 느슨한 검색이 불가능하다.  

<br>

<br>

#### text_any 검색 (문장 단어 1개 이상 포함)

단어에 대해 OR 검색을 하려면 text 대신 text_any를 사용할 수 있다.  

![](/images/img_4be1b5a09baa6759.png)
이건 검색어를 토큰으로 분리한 다음에, 각 토큰이 하나라도 포함된다면 모두 가져온다.  

<br>

<br>

#### phrase 검색 (문장 완전 포함)

보다 엄격한 검색이 필요하다면 phrase를 사용할 수 있다. 이건 현재 검색어가 변형 없이 포함되는 것만 검사한다.  
그리고 이 기능을 사용하려면 인덱스에 먼저 옵션을 활성화해야 한다.  

![](/images/img_18558f743369bc0d.png)

![](/images/img_4a94ae1c162637c7.png)
다른 text 검색은 순서와 상관없이 개별 단어의 일치 여부만 보지만  

<br>

![](/images/img_ff96d77173465c39.png)

![](/images/img_b83efbb8d96d1d26.png)
phrase는 위처럼 순서를 꼬는 것이 통하지 않는다. 검색어의 문장이 정확하게 포함되어야 한다.

## Text 인덱스 옵션들

텍스트 검색에 대한 세부적인 튜닝은, 인덱스를 만드는 시점에 설정해야 한다.  

지원하는 전체 옵션 목록은 다음과 같다.  

![](/images/img_c09d3e40a53afca0.png)
lowercase는 대소문자 구분에 대한 옵션이다.  
기본값은 true고, 모든 토큰을 소문자로 저장해서 대소문자를 구분하지 않는 것이 기본 동작이라는 뜻이다.  

ascii_folding은 라틴 변형 문자들을 뭉치는 것에 대한 옵션이다.  
기본값은 false다. 활성화한다면 라틴 문자에서 움라우트나 악센트 등을 제거한 채로 저장한다.  

그리고 stemmer는 텍스트를 토큰화할때 어근 단위로 가공할지를 선택한다.  
예를 들어, shirts라는 토큰이 나오면 그걸 shirt라고 저장하는 것이다.  

기본 설정에서는 어근 추출이 꺼져있기 때문에, bags를 넣더라도 bag과 일치하지는 않아 검색되지 않는다.  

![](/images/img_7869fed67b9fd95d.png)

하지만 어근 설정을 켠다면  

![](/images/img_f492bc2a5659b80c.png)

![](/images/img_5220957dc78b0b26.png)
형태소의 원형을 기반으로 검색하기에 좀 더 유연한 검색이 가능해진다.

## 하이브리드 검색 (Scoring)

text 필터링 또한 scoring에 포함하는 것이 가능은 하나 아주 매끄럽게 되지는 않는다.  

![](/images/img_9d8ba0b1f43c56ca.png)
이 벡터 검색에 텍스트 검색 기반의 score를 더한다고 하면  

<br>

![](/images/img_048cf3163388723d.png)
잘 되기는 된다.  
이 경우에는 텍스트 필터링에 걸린 항목에 기본 1 score가 더해진 것을 볼 수 있다.  

문제는, text, text_any 등이 단어의 일치 정도에 비례한 score를 반환하지 않는다는 것이다. 조건이 하나라도 맞으면 1을 반환하고, 맞지 않으면 아예 0을 반환한다.  

![](/images/img_906a2378678941c4.png)
그래서 완전 일치 텍스트를 넣든, 부분 일치 텍스트를 넣든 위처럼 무조건 고정 점수 1이 반환되고, 유사도를 적절히 조율하는 것이 어렵다.  

이건 현재 시스템 기능의 한계다. 메인테이너한테 물어보니까 score까지 제어하는건 현재로서는 불가능한 요구사항이라고 하더라.  
편법을 쓰면 달성할 수는 있는데, 직접 토큰화를 한다음에 그 표현식을 종합해서 연산하도록 만드는 것이다.  

![](/images/img_928a171f5895763d.png)
이런 식으로 말이다.  
이상적인 방법은 아니지만, 현재로서는 이것 뿐이다.
